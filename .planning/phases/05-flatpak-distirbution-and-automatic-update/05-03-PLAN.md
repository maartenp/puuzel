**1. .github/workflows/release.yml** (per D-12):

CRITICAL: Flatter's `upload-pages-artifact: true` only uploads the OSTree repo it builds — it does NOT include extra files from the workspace root. The `version.txt` file MUST be injected into the pages artifact so that DIST-02 (in-app update check at `https://maartenp.github.io/puuzel/version.txt`) works. Without this, the version check will always 404.

Strategy: Do NOT use Flatter's `upload-pages-artifact`. Instead, let Flatter build the Flatpak and export the OSTree repo to a local directory, then manually add `version.txt` to that directory, and upload the combined artifact using `actions/upload-pages-artifact` ourselves.

```yaml
name: Release

on:
  push:
    tags: ['v*']

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    container:
      image: ghcr.io/andyholmes/flatter/freedesktop:24.08
      options: --privileged
    steps:
      - uses: actions/checkout@v4

      - name: Install flatpak-cargo-generator dependencies
        run: pip install aiohttp toml

      - name: Clone flatpak-builder-tools
        run: git clone --depth 1 https://github.com/flatpak/flatpak-builder-tools.git

      - name: Generate cargo-sources.json from Cargo.lock
        run: |
          python3 flatpak-builder-tools/cargo/flatpak-cargo-generator.py \
            Cargo.lock -o flatpak/cargo-sources.json

      - name: Build Flatpak
        id: flatter
        uses: andyholmes/flatter@main
        with:
          files: flatpak/io.github.maartenp.puuzel.yml
          upload-pages-artifact: false

      - name: Write version.txt into pages directory
        run: |
          echo "${{ github.ref_name }}" | sed 's/^v//' > ${{ steps.flatter.outputs.repo }}/../version.txt
          cp ${{ steps.flatter.outputs.repo }}/../version.txt ${{ steps.flatter.outputs.repo }}/version.txt

      - name: Upload Pages artifact with version.txt included
        uses: actions/upload-pages-artifact@v3
        with:
          path: ${{ steps.flatter.outputs.repo }}

  deploy:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Key decisions and rationale:
- Uses `ghcr.io/andyholmes/flatter/freedesktop:24.08` image (matches SDK 24.08)
- `cargo-sources.json` generated in CI per D-13 (not committed)
- `upload-pages-artifact: false` on Flatter — we handle artifact upload ourselves so we can inject `version.txt` into the OSTree repo directory before uploading. Flatter exposes the repo path via `${{ steps.flatter.outputs.repo }}`.
- `version.txt` is written directly into the repo output directory so it deploys alongside the OSTree repo at the gh-pages root. This ensures `https://maartenp.github.io/puuzel/version.txt` returns the current version (not 404).
- If `steps.flatter.outputs.repo` is not available, the executor should check Flatter's README for the correct output name and adjust. An alternative fallback: set `upload-pages-artifact: true` on Flatter, then use `actions/download-artifact` to download the pages artifact, inject `version.txt`, and re-upload with `actions/upload-pages-artifact`.
- Permissions set at workflow level for pages deployment
- `pip install aiohttp toml` — the generator script requires these Python packages
